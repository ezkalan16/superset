# Behavioral Impact Report — `apispec` 6.6.1 → 6.7.0 (for human review)

This report covers the "Changes to existing functionality" category from
`DEPENDENCY_UPGRADE_REPORT.md` and assesses whether it actually changes the behavior of
this codebase. One change has real behavioral impact.

## Change: `fields.Dict()` without a value type now emits `"additionalProperties": {}`

- **Source:** `apispec` 6.7.0, "Fix handling of `fields.Dict()` with `values` unset" (issue #949).
- **Before (6.6.1):** a bare `fields.Dict()` produced a schema with no `additionalProperties`
  key, e.g. `{"type": "object"}`.
- **After (6.7.0):** the same field produces `{"type": "object", "additionalProperties": {}}`.

Verified empirically against this codebase (via `parameters_json_schema()`):

| Engine spec | Field | 6.6.1 | 6.7.0 |
|---|---|---|---|
| GSheets | `catalog` | `{"type": "object"}` | `{"type": "object", "additionalProperties": {}}` |
| BigQuery | `query` | `{"type": "object"}` | `{"type": "object", "additionalProperties": {}}` |
| Datastore | `query` | `{"type": "object"}` | `{"type": "object", "additionalProperties": {}}` |

### Likely effect

- **Semantically equivalent, more explicit.** `"additionalProperties": {}` is an empty
  schema that matches any value — the same meaning as omitting the key entirely. OpenAPI
  clients, validators, and codegen tools should treat both forms as "an object with
  arbitrary/untyped properties." No API request/response *data* changes; only the
  self-describing schema does.
- **Low risk, but two things a human should confirm:**
  1. **Checked-in `docs/static/resources/openapi.json` is now stale.** Regenerating it with
     `superset update-api-docs` will produce `additionalProperties: {}` on many Dict fields
     (and may also pick up unrelated drift from other dependency bumps such as marshmallow 4).
     This was intentionally **not** regenerated in the upgrade PR to avoid a large, mixed
     diff and because no CI/pre-commit check enforces its freshness. Decide whether to
     regenerate it as a separate, isolated change.
  2. **Downstream OpenAPI consumers / typed clients.** If any consumer generates strict
     types from the spec, the added `additionalProperties: {}` could change a generated type
     from "opaque object" to "object with an index signature." Confirm no downstream client
     treats the two forms differently.
